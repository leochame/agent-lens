# AIFramework 替代 LangChain4j 开发文档

更新时间：2026-03-31（Step 1/5）

## 1. 目标与约束

目标：在当前 TypeScript 项目中，用 `aiframework` 代码完整替代原 `langchain4j` 在任务执行链路中的职责。

约束：
- 代码精简，不做过度抽象。
- 设计模式按需使用，只在降低复杂度时使用。
- 层级清晰：`core -> adapters -> runtime -> loop-scheduler`。

## 3. 已完成替代（代码事实）

1. `langchain4j` 依赖与引用已清零
- `package.json` 无 `langchain4j` 依赖。
- 全仓无 `langchain4j`/`langchain` 代码引用。

2. 主执行链路已完成替代
- 调度入口：`loop-scheduler/engine.ts` -> `AgentRuntime`。
- 运行时编排：`TaskRuntime` + `WorkflowExecutor`。
- 执行能力闭环：
  - command：`adapters/command/zsh-command-runner.ts`
  - model：`adapters/model/*`
  - tool：`runtime/tool-registry.ts`
  - memory：`adapters/memory/file-memory-store.ts`

3. 分层边界已基本建立
- `loop-scheduler` 不直接依赖 `aiframework/core` 类型模块。
- 关键边界测试已存在：
  - `src/loop-scheduler/runtime-boundary.test.ts`
  - `src/aiframework/runtime/core-type-decoupling.test.ts`

## 4. 未完成替代（需要收口）

1. `aiframework/core` 中仍保留一套重复合同类型（P1）
- `src/aiframework/core/model.ts`
- `src/aiframework/core/workflow.ts`
- `src/aiframework/core/memory.ts`
- `src/aiframework/core/tool.ts`

说明：当前 runtime 主链路已不依赖这套类型（仅 `core/task-id.ts` 仍在使用）。这批重复合同会造成双语义入口，后续维护成本偏高。

2. runtime 合同仍保留兼容 alias（P1）
- `src/aiframework/runtime/contracts.ts` 同时导出 `Runtime*` 与旧别名（带 `@deprecated`）。

说明：兼容策略是合理的，但目前还处于过渡态，尚未完成最终收口。

3. scheduler 兼容字段仍存在（P2）
- `loop-scheduler/types.ts` 的 `timeoutSec` 仍保留在输入兼容层。

说明：执行语义已统一为 runtime timeout disabled，但字段还在，需按窗口完成退场。

4. `LoopScheduler` Facade 仍较重（P2）
- `src/loop-scheduler/engine.ts` 已拆分过，但仍集中管理任务生命周期、队列、运行态、持久化回写。

说明：当前可维护，但继续增加功能时有膨胀风险。

## 5. Step 1 结论

结论：
- “用 `aiframework` 替代 `langchain4j` 主链路”已经完成。
- 当前工作重心不是“能不能替代”，而是“替代后的结构收口与长期可维护性”。

当前依赖方向（符合目标）：
- `loop-scheduler -> aiframework/runtime -> aiframework/adapters`
- `runtime` 仅保留对 `core/task-id.ts` 的工具型依赖

## 6. Step 2-5 开发计划

### Step 2/5（P1）：清理 core 重复合同，收敛单一合同源
目标：只保留一套运行时合同，避免双语义。

实施：
- 将 `src/aiframework/core/{model,workflow,memory,tool}.ts` 标记为 legacy（或迁移为 runtime 直接导出）。
- 清理仓库内对上述文件的潜在引用入口（含测试和文档）。
- 保留 `core/task-id.ts` 作为纯工具模块。

验收：
- `rg -n "aiframework/core/(model|workflow|memory|tool)" src -S` 无生产代码引用。
- 全量测试通过。

### Step 3/5（P1）：推进 runtime alias 退场（第一阶段）
目标：新代码只使用 `Runtime*` 命名，兼容别名只保留最小表面积。

实施：
- 统一运行时代码与测试到 `Runtime*` 类型名。
- 对仍需 alias 的位置增加明确退场注释和时间窗口。

验收：
- `runtime/contracts.ts` 之外不再新增 alias 使用点。
- 全量测试通过。

### Step 4/5（P2）：继续减轻 LoopScheduler 体量（按职责微拆分）
目标：保持 Facade，但降低单类复杂度。

实施：
- 提取不改变外部 API 的内部协作者（例如运行回写、队列泵、取消控制）。
- 严格避免过度抽象，优先函数级拆分。

验收：
- `engine.ts` 复杂度下降（函数/行数/分支减少）。
- 行为回归测试通过。
